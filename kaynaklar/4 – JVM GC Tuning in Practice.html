Bölüm 4 – JVM GC Tuning in Practice

GC Nedir, Nasıl Çalışır?

Garbage Collection (GC), JVM'in kullanılmayan nesneleri otomatik olarak temizlemesi ve heap bellek alanını yönetmesi sürecidir. Java'da geliştirici bellek tahsisini yapar ancak serbest bırakma JVM tarafından otomatik olarak yapılır.

GC Çalışma Mantığı:

  • Minor GC: Young Generation'daki nesneleri temizler. Genellikle hızlıdır.
  • Major GC / Full GC: Old Generation'daki nesneleri temizler. Uygulamayı durdurur (Stop-The-World), performansı ciddi etkiler.
  • Mixed GC: G1GC'de hem young hem de bazı old region'ları temizler.

🎯 Performansa Etkisi:

GC süreçleri sırasında JVM Stop-The-World (STW) duraklamaları yaşar. Bu duraklamalar REST API'lerde response time artışlarına ve throughput düşüşlerine neden olur. Özellikle yüksek trafikli sistemlerde GC tuning kritik önem taşır.

GC Log Formatı ve Analizi

GC logları, JVM'in bellek yönetimini anlamak ve performans sorunlarını teşhis etmek için en önemli veri kaynağıdır.

[0.038s][info][gc] Using G1
[0.839s][info][gc] GC(0) Pause Young (G1 Evacuation Pause) 18M->2M(64M) 3.214ms
[2.157s][info][gc] GC(1) Pause Young (G1 Evacuation Pause) 20M->3M(64M) 4.892ms
[5.782s][info][gc] GC(2) Pause Full (System.gc()) 50M->10M(64M) 120.456ms

Log Formatı Açıklaması:

  • [0.839s]: Uygulama başlangıcından itibaren geçen süre
  • GC(0): GC olayının sırası
  • Pause Young (G1 Evacuation Pause): GC türü ve nedeni
  • 18M->2M(64M): GC öncesi heap kullanımı → GC sonrası heap kullanımı (toplam heap)
  • 3.214ms: GC süresi (Stop-The-World süresi)

🎯 Performansa Etkisi:

GC log analizi yapmadan tuning yapmak, kör bir şekilde araba sürmek gibidir. Uzun GC süreleri (>100ms) REST API response time'ı doğrudan etkiler. Özellikle e-ticaret, finans gibi gerçek zamanlı sistemlerde 100ms üzeri GC duraklamaları kabul edilemez.

G1GC, ZGC, Shenandoah Farkları

GC Türü Max Pause Time Heap Boyutu JDK Sürümü Kullanım Senaryosu
G1GC ~200ms 6GB+ JDK 7u4+ Genel amaçlı, dengeli performans
ZGC <10ms 8TB JDK 11+ Düşük gecikme, büyük heap
Shenandoah <10ms Yok JDK 12+ Düşük gecikme, küçük-medium heap

Temel Farklar:

  • G1GC: Region tabanlı, pause time hedeflenebilir (-XX:MaxGCPauseMillis), ancak büyük heap'lerde pause time artar.
  • ZGC: Concurrent compaction, pause time heap boyutundan bağımsız, büyük heap'ler için ideal.
  • Shenandoah: Concurrent marking ve evacuation, pause time çok düşük, ancak CPU overhead yüksek.

🎯 Performansa Etkisi:

ZGC ve Shenandoah, G1GC'ye göre 10-20x daha düşük pause time sağlar. Bu, REST API'lerde response time tutarlılığını artırır. Ancak CPU tüketimi %10-20 artabilir. Trafik yoğunluğu yüksek, latency kritik sistemlerde ZGC/Shenandoah tercih edilmelidir.

Clustered Uygulamalarda GC Tuning Zorlukları

Dağıtık Sistemlerde GC Problemleri:

  • Eşzamanlı GC Duraklamaları: Cluster'deki tüm node'lar aynı anda GC yaparsa servis tamamen çöker.
  • Network Timeout: GC sırasında node'lar heartbeat gönderemez, load balancer node'u devre dışı bırakır.
  • Cache Invalidation: GC uzun sürerse distributed cache tutarsızlığı oluşur.
  • Database Connection Timeout: GC sırasında DB bağlantıları timeout olur, transaction rollback olur.

🎯 Performansa Etkisi:

Clustered ortamlarda GC tuning yapılmazsa, sistem genelinde cascade failure (domino etkisi) oluşur. Tek bir node'un GC'si tüm cluster'ı etkileyebilir. Bu nedenle GC pause time'ların 10ms altına çekilmesi ve GC zamanlamasının randomize edilmesi kritiktir.

⚠️ Çözüm Önerileri:

  • -XX:+UnlockExperimentalVMOptions -XX:ZCollectionInterval=30 ile GC'leri eşzamansızlaştır
  • Health check endpoint'lerde GC durumunu kontrol et
  • Load balancer timeout sürelerini GC pause time + α olarak ayarla

💻 Demo Senaryosu: Spring Boot API ile GC Karşılaştırması

Aynı Spring Boot 3.5.5 REST API uygulamasını farklı GC algoritmalarıyla çalıştırarak performans karşılaştırması.

1. Uygulama Hazırlığı

@RestController
public class PerformanceController {
    
    // 10MB boyutunda String nesneleri yaratıyoruz
    private List cache = new ArrayList<>();
    
    @GetMapping("/test-gc")
    public ResponseEntity testGC() {
        // Her istekte 10MB veri üretiyoruz (GC tetiklemek için)
        String largeData = "X".repeat(10_000_000);
        cache.add(largeData); // Memory leak simülasyonu
        
        if (cache.size() > 100) {
            cache.clear(); // Manuel temizlik (GC tetiklemek için)
        }
        
        return ResponseEntity.ok("GC Test Completed - Cache Size: " + cache.size());
    }
}

2. G1GC ile Çalıştırma

java -Xmx512m -Xms512m \
-XX:+UseG1GC \
-XX:MaxGCPauseMillis=200 \
-XX:+PrintGCDetails \
-XX:+PrintGCDateStamps \
-Xlog:gc*:file=gc-g1.log:time \
-jar spring-boot-app.jar

3. ZGC ile Çalıştırma

java -Xmx512m -Xms512m \
-XX:+UseZGC \
-XX:+PrintGCDetails \
-XX:+PrintGCDateStamps \
-Xlog:gc*:file=gc-zgc.log:time \
-jar spring-boot-app.jar
