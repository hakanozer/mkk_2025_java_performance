Bölüm 2: Profiling Tools & JVM Profiling

📊Profiling Nedir?

Profiling, uygulamanızın çalışma zamanında performans metriklerini toplama ve analiz etme sürecidir. Bu süreç, performans sorunlarının kök nedenlerini belirlememizi sağlar.

🎯 Profiling'in Temel Amaçları:
  • CPU kullanımını analiz etmek
  • Memory leak'leri tespit etmek
  • Thread contention'ları bulmak
  • I/O darboğazlarını identifiye etmek
  • Garbage Collection performansını optimize etmek
⚡ Performans Etkisi: Profiling araçları uygulamanın performansını %5-15 oranında etkileyebilir. Production ortamında dikkatli kullanılmalıdır.

Profiling Türleri

  • Sampling Profiler: Belirli aralıklarla stack trace alır (düşük overhead)
  • Instrumentation Profiler: Her metod çağrısını izler (yüksek overhead)
  • Hybrid Profiler: Her iki yöntemi kombinler

Ej-technologies tarafından geliştirilen ticari profiling aracıdır.

JProfiler Avantajları:
  • Gelişmiş GUI ve raporlama
  • Database profiling
  • Probes ve triggers
  • Production profiling desteği
  • Detaylı call tree analizi

❌ Profiling Öncesi

  • Performans sorunları tahmin edilir
  • Optimizasyon kararları varsayımlara dayanır
  • Memory leak'ler production'da keşfedilir

✅ Profiling Sonrası

  • Performans sorunları veri ile desteklenir
  • Optimizasyon hedeflenir ve ölçülür
  • Memory pattern'leri görselleştirilir
  • CPU kullanımı detaylı analiz edilir

⚙️JVM Flags: -Xms, -Xmx, -XX:+UseG1GC

🎯 Kritik JVM Parametreleri: JVM'in doğru yapılandırılması, uygulamanın performansını 2-5 kata kadar artırabilir.

Memory Ayarları

application.properties
# Heap Size Ayarları -Xms2g # Initial heap size (2GB) -Xmx8g # Maximum heap size (8GB) -Xmn2g # Young generation size (2GB) # MetaSpace Ayarları -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=1g # Direct Memory -XX:MaxDirectMemorySize=2g

Garbage Collector Seçimi

GC Algoritmaları
# G1 Garbage Collector (Önerilen) -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:G1HeapRegionSize=16m # Parallel GC (High Throughput) -XX:+UseParallelGC -XX:ParallelGCThreads=8 # ZGC (Ultra-Low Latency - Java 15+) -XX:+UseZGC -XX:+UnlockExperimentalVMOptions
⚡ Performans Karşılaştırması:
  • Default GC: 50-200ms pause times
  • G1GC: 10-50ms pause times
  • ZGC: <1ms pause times
⚠️ Dikkat: Heap size'ı sistem RAM'inin %70'inden fazla olmamalıdır. İşletim sistemi ve diğer süreçler için yer bırakın.

🗃️Heap Dump ve Thread Dump Alma

Heap Dump Nedir?

Belirli bir anda JVM heap'inin anlık görüntüsüdür. Memory leak'leri ve object'lerin memory kullanımını analiz etmek için kullanılır.

Heap Dump Alma Yöntemleri
# jcmd ile (Java 8+) jcmd <pid> GC.run_finalization jcmd <pid> VM.gc jcmd <pid> GC.dump filename=heapdump.hprof # jmap ile (Eski yöntem) jmap -dump:live,format=b,file=heapdump.hprof <pid> # Otomatik heap dump (OOM durumunda) -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dumps/

Thread Dump Nedir?

Tüm thread'lerin o anki durumunu gösteren snapshot'tır. Deadlock'ları ve thread contention'ları tespit etmek için kullanılır.

Thread Dump Alma
# jstack ile jstack <pid> > threaddump.txt # jcmd ile (Tercih edilen) jcmd <pid> Thread.print > threaddump.txt # kill signal ile (Linux/Mac) kill -3 <pid> # VisualVM ile GUI üzerinden # JConsole ile MBeans tab -> HotSpotDiagnostic -> dumpThreads
🎯 Thread Dump Analiz Araçları:
  • Eclipse MAT: Memory Analyzer Tool
  • IBM Thread and Monitor Dump Analyzer: IBM'in ücretsiz aracı
  • FastThread: Online thread dump analizi
  • VisualVM: Built-in thread viewer

Custom Metrics Ekleme

MetricsController.java
@RestController @RequestMapping("/api") public class MetricsController { private final MeterRegistry meterRegistry; private final Counter apiCallCounter; private final Timer responseTimer; public MetricsController(MeterRegistry meterRegistry) { this.meterRegistry = meterRegistry; this.apiCallCounter = Counter.builder("api.calls") .description("Total API calls") .register(meterRegistry); this.responseTimer = Timer.builder("api.response.time") .description("API response time") .register(meterRegistry); } @GetMapping("/users") @Timed(value = "api.users.timer", description = "Time taken to get users") public ResponseEntity<List<User>> getUsers() { Timer.Sample sample = Timer.start(meterRegistry); try { apiCallCounter.increment(); // Business logic here List<User> users = userService.getAllUsers(); return ResponseEntity.ok(users); } finally { sample.stop(responseTimer); } } }
⚡ Actuator Performans Etkisi:
  • Endpoint'lerin aktif edilmesi: ~1-2% CPU overhead
  • Metrics collection: ~0.5-1% memory overhead
  • Health check'ler: Minimal etki

🔄 Demo 3: Sonsuz Döngü → Thread Dump Analizi

InfiniteLoopController.java
@RestController @RequestMapping("/api/thread") public class InfiniteLoopController { @GetMapping("/infinite-loop") public ResponseEntity<String> infiniteLoop() { // Sonsuz döngü simülasyonu while (true) { try { // CPU'yu %100 kullanmasın diye küçük bir sleep Thread.sleep(1); } catch (InterruptedException e) { Thread.currentThread().interrupt(); break; } } return ResponseEntity.ok("Bu response asla dönmez!"); } @GetMapping("/deadlock") public ResponseEntity<String> deadlock() { Object lock1 = new Object(); Object lock2 = new Object(); Thread t1 = new Thread(() -> { synchronized (lock1) { try { Thread.sleep(100); } catch (InterruptedException e) {} synchronized (lock2) { System.out.println("Thread 1: Acquired both locks"); } } }); Thread t2 = new Thread(() -> { synchronized (lock2) { try { Thread.sleep(100); } catch (InterruptedException e) {} synchronized (lock1) { System.out.println("Thread 2: Acquired both locks"); } } }); t1.start(); t2.start(); return ResponseEntity.ok("Deadlock başlatıldı!"); } }
Thread Dump Analizi
# 1. Infinite loop endpoint'ini çağırın (başka bir terminal'de) curl http://localhost:8080/api/thread/infinite-loop & # 2. Thread dump alın jcmd <PID> Thread.print > infinite-loop-threaddump.txt # 3. Analiz edin - RUNNABLE durumunda takılan thread'i bulun grep -A 20 "infinite-loop" infinite-loop-threaddump.txt # Deadlock testi için: curl http://localhost:8080/api/thread/deadlock jcmd <PID> Thread.print > deadlock-threaddump.txt # Deadlock detection: grep -A 10 "Found Java-level deadlock" deadlock-threaddump.txt
📊 Thread Dump Analiz Kriterleri:
  • RUNNABLE: Aktif çalışan thread'ler
  • BLOCKED: Monitor lock bekleyen thread'ler
  • WAITING: wait(), join() çağrısı yapan thread'ler
  • TIMED_WAITING: sleep(), wait(timeout) durumundaki thread'ler
